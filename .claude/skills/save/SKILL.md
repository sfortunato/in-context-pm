# /save — Ship markdown docs in one command

Commit markdown changes, open a PR, and auto-merge after review — all in one command. A background agent handles the review cycle so the user can keep working.

**This skill only operates on `.md` files.** It is designed for PMs and non-engineers who edit documentation and want to ship changes without babysitting git or PR reviews.

## Step 1: Check for changes

Run `git status --porcelain` to check for uncommitted changes (staged, unstaged, and untracked files).

**If there are no changes at all:**
- Say: "You're all good here. Nothing to save."
- Stop.

## Step 2: Enforce markdown-only guard

From the `git status` output, extract every changed/untracked file path. Check that EVERY file matches `*.md`.

**If ANY non-markdown files have changes (staged, unstaged, or untracked):**
- Say: "/save only works for markdown-only changes. You have changes to non-markdown files — use a regular commit workflow instead."
- List the non-markdown files so the user can see what's blocking.
- Stop.

## Step 3: Detect existing branch or PR

Determine the repo's default branch and check what branch you're on:

```bash
DEFAULT_BRANCH=$(gh repo view --json defaultBranchRef --jq .defaultBranchRef.name 2>/dev/null || echo "main")
CURRENT_BRANCH=$(git branch --show-current)
echo "Default: $DEFAULT_BRANCH, Current: $CURRENT_BRANCH"
```

**If on the default branch:** proceed to Step 4.

**If on a feature branch:**
1. Check if an open PR exists: `gh pr view --json number,url 2>/dev/null`
2. If a PR exists: ask the user via AskUserQuestion — "You're on branch X with an open PR (URL). Do you want to add your new changes to this PR, or start fresh on a new branch?"
   - If "add to existing": stage and commit the `.md` files to the current branch, push, then skip to Step 6 (spawn background agent for this PR).
   - If "start fresh": `git stash`, `git checkout $DEFAULT_BRANCH`, `git pull`, then proceed to Step 4.
3. If no PR exists: ask the user — "You're on branch X but there's no PR. Do you want to open a PR for this branch, or switch to $DEFAULT_BRANCH and start fresh?"
   - If "open PR for this branch": stage and commit, push, create PR via `gh pr create`, then proceed to Step 6.
   - If "start fresh": `git stash`, `git checkout $DEFAULT_BRANCH`, `git pull`, then proceed to Step 4.

## Step 4: Create branch, commit, and push

1. Get the branch name components:
   ```bash
   whoami
   date +%m-%d
   ```
   Branch format: `<user>/<month-day>-<summary>` — infer a short summary from the changed `.md` file names.

2. Stash, pull, create branch:
   ```bash
   git stash
   git pull origin "$DEFAULT_BRANCH"
   git checkout -b <branch-name>
   git stash pop
   ```

3. Stage only `.md` files and commit:
   ```bash
   git add <list of .md files>
   ```
   Auto-generate the commit message:
   - Single file: "Update <filename>" (e.g., "Update user-onboarding product spec")
   - Multiple files: "Update docs: <brief summary of all changed files>"

4. Push with upstream tracking:
   ```bash
   git push -u origin <branch-name>
   ```

## Step 5: Open a PR

Create the PR targeting the default branch:

```bash
gh pr create --title "<auto-generated title>" --body "<auto-generated description listing changed files>"
```

The title should match the commit message. The body should list the changed files.

Print the PR URL to the user.

## Step 6: Spawn background agent

Tell the user: "Watching your PR in the background — I'll let you know when it merges (or if something needs your attention)."

Use the **Agent tool** with `run_in_background: true` to spawn a background agent. Pass the agent a complete, self-contained prompt with all the context it needs (PR number, owner/repo, branch name, and the full instructions below). The background agent does NOT have access to this SKILL.md file, so the prompt must contain everything.

### Background agent prompt

Include ALL of the following in the agent prompt, substituting the actual PR number, owner, repo, and branch:

---

**Your job:** Watch PR #NUMBER on OWNER/REPO (branch: BRANCH) until it merges or you hit a problem you can't fix. This is a documentation-only PR (markdown files). Be autonomous — fix what you can, pause and notify the user only when you're stuck.

**Phase 1: Wait for CI**

Poll for CI status every 60 seconds, up to 5 minutes:

```bash
gh pr checks NUMBER --repo OWNER/REPO --json name,state,conclusion \
  --jq '.[] | select(.conclusion != "" and .conclusion != "success" and .conclusion != "neutral" and .conclusion != "skipped") | .name + ": " + .conclusion'
```

- If all checks pass or no checks are configured: proceed to Phase 2.
- If any check fails: notify the user: "CI check failed on your docs PR: [details]. You may need to take a look." Include the PR link. Stop.
- Print a brief status update every 2 minutes so the user knows you're still watching.

**Phase 2: Fix review comments**

Track fix attempts. Maximum 3 fix-commit cycles.

For each cycle:

1. Fetch ALL unresolved review threads using the GitHub GraphQL API. Write the query to a temp file — inline queries break with special characters.

   ```bash
   cat <<'QUERY_EOF' > /tmp/pr_threads.graphql
   query($owner: String!, $repo: String!, $pr: Int!, $cursor: String) {
     repository(owner: $owner, name: $repo) {
       pullRequest(number: $pr) {
         reviewThreads(first: 100, after: $cursor) {
           totalCount
           pageInfo { hasNextPage endCursor }
           nodes {
             id
             isResolved
             comments(first: 50) {
               nodes {
                 author { login }
                 body
                 path
                 line
                 createdAt
                 databaseId
               }
             }
           }
         }
       }
     }
   }
   QUERY_EOF
   gh api graphql -F query=@/tmp/pr_threads.graphql -f owner=OWNER -f repo=REPO -F pr=NUMBER
   ```

   **CRITICAL:** Always check `hasNextPage`. If true, fetch subsequent pages using `endCursor` as the cursor variable. Collect ALL pages before proceeding.

2. If no unresolved threads: proceed to Phase 3.

3. For each unresolved thread:
   - Read the FULL comment chain (not just the first comment — check for follow-up replies)
   - Read the referenced file to understand the current state
   - Attempt to apply the suggested fix
   - If the fix succeeds: stage the change
   - If the fix fails (can't parse suggestion, file doesn't exist, change would break structure): mark this thread as unfixable

4. If ANY threads are unfixable after a genuine attempt:
   - Notify the user: "A reviewer left a comment I couldn't fix on your docs PR. Here's what it said: [summary]. PR link: [URL]"
   - Stop. Leave the PR open for manual resolution.

5. If all threads were fixed: commit all fixes together:
   ```bash
   git add <fixed files>
   git commit -m "Address review feedback"
   git push
   ```

6. Reply to each thread explaining the fix:
   ```bash
   gh api repos/OWNER/REPO/pulls/NUMBER/comments/COMMENT_DATABASE_ID/replies \
     -f body="Fixed — [brief description of change]" --method POST
   ```

7. Resolve each thread:
   ```bash
   cat <<'RESOLVE_EOF' > /tmp/resolve_thread.graphql
   mutation($threadId: ID!) {
     resolveReviewThread(input: {threadId: $threadId}) {
       thread { isResolved }
     }
   }
   RESOLVE_EOF
   gh api graphql -F query=@/tmp/resolve_thread.graphql -f threadId=THREAD_NODE_ID
   ```

8. Wait 20 seconds for review tools to process the new commit, then re-fetch threads (go back to step 1 of this phase).

9. If this is the 3rd fix cycle and new comments keep appearing: notify the user: "I've fixed review comments 3 times but new ones keep appearing. You may want to take a look. PR link: [URL]". Stop.

**Phase 3: Merge**

When all threads are resolved and CI is green:

```bash
gh pr merge NUMBER --repo OWNER/REPO --squash --delete-branch
```

Notify the user: "Your changes are merged! PR_URL"
